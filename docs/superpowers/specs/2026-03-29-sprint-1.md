## Issues

### #1 — Consolidate text sanitization into parse_message()

**Priority:** High (foundational — other issues depend on clean parsing)

**Problem:** Sanitization is split across `parsing.py` (`str()` coercion) and `parse.py` (`_sanitize_row()`). Downstream code can't trust that `parse_message()` returns clean, database-safe strings.

**Spec:**
- Add `_safe_header(msg, name) -> str | None` helper in `parsing.py` that:
  - Calls `msg.get(name)`
  - Coerces `email.header.Header` objects to `str`
  - Strips NUL bytes (`\x00`)
  - Handles encoding errors gracefully
- Use `_safe_header()` for all header access in `parse_message()`
- Remove `_sanitize_row()` from `parse.py` once all sanitization is at the parse layer

**Files:**
- `src/maildb/parsing.py`
- `src/maildb/ingest/parse.py`

**Tests:** Unit tests for `_safe_header()` covering Header objects, NUL bytes, encoding errors, None values.

### #2 — Per-message error handling in chunk processing

**Priority:** High (data loss prevention)

**Problem:** `_process_single_chunk()` inserts all rows in one transaction. One bad row fails the entire chunk (hundreds of messages).

**Spec:**
- Wrap per-row INSERT in try/except inside `_process_single_chunk()`
- On failure: log error with message_id, skip the row, continue
- Track skipped count alongside inserted/duplicate counts
- Chunk is still marked complete with accurate stats

**Files:**
- `src/maildb/ingest/parse.py` — `_process_single_chunk()`

**Tests:** Integration test with a deliberately malformed message in a chunk to verify the chunk completes with correct skip count.

### #3 — Validate recipients JSON structure before insert

**Priority:** Medium

**Problem:** `parse_message()` constructs recipients dict from `email.utils.getaddresses()` without validating output. Malformed headers could produce unexpected structures that break JSONB insert or downstream queries.

**Spec:**
- Validate in `parse_message()` that:
  - `recipients` is a dict with keys `to`, `cc`, `bcc`
  - Each value is a list of strings
  - Empty/None addresses are filtered out
- Use a Pydantic model or simple defensive check

**Files:**
- `src/maildb/parsing.py` — `parse_message()`

**Tests:** Unit tests with malformed recipient headers (missing fields, None addresses, non-string values).

---

### #4 — Token-aware embedding truncation

**Priority:** Medium (improves embedding quality)

**Problem:** The 6K char limit is safe but wastes context for normal English emails. Char-to-token ratio varies: ~4 chars/token for prose vs ~1.2 for URLs/code.

**Spec:**
- Replace char-based truncation with token-aware approach
- Use fast token count approximation (e.g., byte length / 4, or lightweight tokenizer)
- Truncate at ~7500 tokens to leave headroom below the 8192 limit
- Fall back to the 6K char limit if no tokenizer is available

**Files:**
- `src/maildb/embeddings.py` — `build_embedding_text()`, `MAX_EMBEDDING_CHARS`

**Tests:** Unit tests with English prose (should allow more than 6K chars), URL-heavy text (should truncate earlier), and edge cases.

### #6 — Make embed phase incrementally resumable

**Priority:** High (operational reliability)

**Problem:** Orchestrator checks `embed_status["completed"] == 0` to decide whether to run embed. Once marked complete, won't re-run even if rows have `embedding IS NULL`. Requires manual SQL reset.

**Spec:**
- Change orchestrator to check for remaining un-embedded rows:
  ```python
  unembedded = count_unembedded(pool)  # SELECT count(*) WHERE embedding IS NULL
  if unembedded > 0:
      # Run embed workers regardless of task status
  ```
- Makes embed phase truly idempotent and resumable

**Files:**
- `src/maildb/ingest/orchestrator.py` — `run_pipeline()` embed phase gate

**Tests:** Integration test: embed partially, stop, re-run — verify remaining rows get embedded.

### #7 — Add `reset` CLI command for ingest pipeline

**Priority:** Medium

**Problem:** Resetting pipeline state requires raw SQL. Error-prone for partial resets.

**Spec:**
- Add `reset` subcommand:
  ```bash
  python -m maildb.ingest reset              # Full reset
  python -m maildb.ingest reset --phase embed # Reset only embed phase
  python -m maildb.ingest reset --phase parse # Reset parse + downstream
  ```
- Each reset level: clear relevant ingest_tasks, clear/NULL relevant data
- Confirm before proceeding (unless `--yes` is passed)

**Files:**
- `src/maildb/ingest/__main__.py` — add `reset` command
- `src/maildb/ingest/orchestrator.py` — add reset functions

**Tests:** Integration tests for each reset level verifying correct data is cleared.
